## Glossary

- **KEEP** — Scenario describes intended behavior that still exists in the v3 codebase. No test asserts this scenario yet. Phase 3 binding waves will write the test and add `@scenario` JSDoc.
- **UPDATE** — The scenario as-written diverges from the actual implementation (changed labels, changed semantics, renamed concepts). Rewrite the scenario before binding, or fix the implementation if the scenario was right and the code drifted.
- **DELETE** — Aspirational, stale, or describes behavior that was never built or has since been removed. Cut from the spec in Phase 1.
- **DUPLICATE** — Already covered by another scenario in this directory, by an existing test, or by a sibling component test. Phase 1 removes the spec scenario and (where useful) leaves a cross-link in the rationale of the duplicate row.

## Audit methodology

- 5 parallel sub-agents read every assigned `.feature` file plus the relevant source under `langwatch/src/evaluations-v3/`, `langwatch/src/server/evaluations-v3/`, `langwatch/src/app/api/evaluations/`, and `langwatch/src/pages/[project]/evaluations/`.
- Default for genuinely uncertain calls: KEEP (with a flag in the rationale). Phase 1/3 reviewers may flip to UPDATE/DUPLICATE on closer reading.
- Phase 1 cull soldiers should treat DUPLICATE rationales as the primary source of cross-links — they cite the test or sibling scenario that already covers the behavior.
