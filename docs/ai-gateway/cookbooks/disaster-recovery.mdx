This cookbook answers the enterprise procurement question: **"If LangWatch goes down, do we lose our AI?"** Short answer: no, not for a long time — the gateway is designed to keep serving traffic across every plausible single-component failure. This page documents each scenario, how long you've got before degradation, and how to intervene.

Pairs with [Production runbook](/ai-gateway/cookbooks/production-runbook) (for p99 / leak / memory incidents on a healthy gateway) and [Prometheus alerts](/ai-gateway/cookbooks/prometheus-alerts) (for the signals that tell you a DR scenario is active).

## Failure scenarios at a glance

| Scenario | Customer impact | How long gateway stays up | First operator action |
|---|---|---|---|
| LangWatch control plane down | None (for most workloads) | ≥ 15 min (JWT cache) + potentially hours (bootstrap cache) | Restore control plane; check outbox shelf |
| Gateway pod killed | None | N/A (HPA replaces) | None |
| One upstream provider down | None for VKs with fallback; 502s for VKs without | Indefinite | Verify fallback fired; provider status |
| Redis L2 cache down | ~30 ms added latency on new-VK resolves | Indefinite | Restore Redis; L1 continues working |
| Gateway region down | Customer-facing; needs DNS failover | Immediate | Failover traffic to surviving region |
| Both gateway regions down | Customer-facing | 0 | Rebuild — see last section |

## Scenario 1: LangWatch control plane is unreachable

**What's broken:** `app.langwatch.ai` is down. Gateway cannot reach `/api/internal/gateway/resolve-key`, `/config/:vk_id`, `/changes`, `/budget/debit`, or `/guardrail/check`.

**What still works:**

- **Existing VK traffic, in full**, for the duration of the gateway's JWT cache TTL (15 min default, `LW_GATEWAY_AUTH_CACHE_TTL_SECONDS`). Every VK that made a request in the last 15 min is still in L1 cache and continues to authenticate instantly.
- **Bootstrap-pulled VKs** — if `LW_GATEWAY_AUTH_CACHE_BOOTSTRAP=true` (default), every active VK in every org known to this gateway was loaded into L1 on startup. These stay resolved until their individual JWT expires (15 min from issue).
- **Redis L2 cache** (iter 15) — if enabled, even VKs not in L1 can warm from Redis without a control-plane round-trip. ~2 ms/request. Gives you hours of headroom on VKs that weren't explicitly pre-cached.
- **Budget debits** — the gateway's outbox (iter 21) re-enqueues failed debits with exponential backoff. `gateway_budget_debit_outbox_flush_failures_total` rises but debits are safe; they flush automatically when the control plane returns.
- **Guardrails** — if VK has `guardrails.{request,response}_fail_open: true`, guardrail calls that fail are passed through. Default is fail-closed (safer but brittle under outage); flip per VK if availability trumps policy.

**Degradation timeline:**

| Elapsed | What starts failing |
|---|---|
| 0 – 15 min | Nothing for cached VKs |
| 15 min+ | First VK hits JWT expiry → cache miss → `resolve-key` fails → **first new 503 `upstream_timeout` on previously-working VKs** if L2 also cold |
| 15 min – N hours | Traffic gradually degrades as each VK's cache entry expires; VKs re-resolving from L2 (if enabled) still fine |
| Outbox ring full | **Silent data loss via `gateway_budget_debit_outbox_dropped_total`** — see [Production runbook Recipe 6](/ai-gateway/cookbooks/production-runbook#recipe-6--debit-outbox-backlog) |

**Operator actions:**

1. Restore the control plane. Once `/api/internal/gateway/*` returns 2xx, the gateway automatically resumes — `/changes` long-poll reconnects, outbox drains, `resolve-key` on next cache miss succeeds.
2. Watch `gateway_budget_debit_outbox_depth` drain back to ~0 over the next few minutes.
3. Audit any VKs created or revoked during the outage — their state may have advanced on the control plane while the gateway couldn't see the `/changes` feed, so a brief window of gateway-serving-revoked-VK is possible. Restart gateway pods if you need immediate convergence.

**What makes this survivable:** the gateway is designed to be a *cache* of the control plane, not a client. It can serve traffic for the cache's lifetime without ever talking to LangWatch.

## Scenario 2: Gateway pod killed or crashed

**What's broken:** one replica terminated unexpectedly (OOM / node eviction / SIGKILL after stuck drain).

**Impact:** none. The gateway is stateless; HPA + PDB ensure ≥ 2 replicas. Traffic routes to surviving pods, the new pod warms from Redis L2 (~2 ms/VK) + `/bootstrap` + `/changes`.

**Operator action:** only investigate if a specific pod is cycling. Use [Production runbook Recipe 3 (memory growth)](/ai-gateway/cookbooks/production-runbook#recipe-3--memory-growth) or [Recipe 7 (stuck drain)](/ai-gateway/cookbooks/production-runbook#recipe-7--stuck-drain) depending on the crash mode.

**What makes this survivable:** everything persistent lives on the control plane; the gateway only holds JWTs + compiled RE2 patterns + in-flight requests. Any of these reconstruct in milliseconds.

## Scenario 3: One upstream provider is down

**What's broken:** OpenAI returning 5xx, or Anthropic returning TCP resets, or a specific Bedrock region timing out.

**What still works:**

- **VKs with fallback chains** — the circuit breaker (iter 4) trips after 10 failures in 30 s, opens for 60 s, and during that window every request on this VK falls over to the next chain credential (5xx / timeout / rate-limit / network triggers). Transparent. See [Fallback chains](/ai-gateway/providers/fallback-chains).
- **VKs without fallback** — return `502 provider_error` or `504 upstream_timeout` to the caller with the correct `X-LangWatch-Provider` header. Clients can retry with their own backoff.

**Degradation:**

| VK config | Customer impact |
|---|---|
| fallback chain configured | Invisible — spans still show fallback events, circuit metric goes from 0 to 2 |
| no fallback | 502 / 504 returned to client |

**Operator actions:**

1. Check the provider status page — this is almost always the answer.
2. Grafana dashboard's "Circuit state by credential" panel shows which VKs/credentials are degraded.
3. If provider has recovered but circuits are still open, they auto-probe every 60 s and self-heal. No manual reset needed.
4. If one *region* of a provider is specifically degraded and the VK's fallback includes a different region of the same provider, that's the cleanest path. Consider whether your fallback chains cross regions.

**What makes this survivable:** the circuit breaker + fallback engine is in the hot path, pre-bifrost, adding ~64 + 119 ns per request. You can rely on it under 10K+ req/s without concern.

## Scenario 4: Redis L2 cache is unreachable

**What's broken:** `LW_GATEWAY_REDIS_URL` points at a Redis that's down, flapping, or unreachable.

**What still works:**

- **L1 cache** — unaffected. Every VK already in the process-local LRU continues to serve.
- **New-VK resolves** — fall through to `/api/internal/gateway/resolve-key` on the control plane. Adds ~30 ms to the first request from a new-to-this-pod VK vs ~2 ms with Redis working.
- **Gateway bootstrap** — `/bootstrap` on startup doesn't use Redis, so a new pod coming up with Redis down is fine (just slower on cold VKs until the L1 warms).

**Degradation:**

| When | Impact |
|---|---|
| Immediate | ~30 ms added latency per cold-for-this-pod VK |
| New pod scale-up | "Thundering herd" of `/resolve-key` calls as all cold VKs miss both layers (but control plane scales sub-linearly, so this is usually fine — see [Scaling → Control-plane capacity](/ai-gateway/self-hosting/scaling#control-plane-capacity)) |

**Operator actions:**

1. Restore Redis. Gateway reconnects automatically; next cache writes go back to populating L2.
2. If Redis will be down for extended period, consider temporarily bumping gateway replicas so more of the working-set stays in L1 cache across the fleet.

**What makes this survivable:** the L2 cache is a performance optimization, not a correctness requirement. The fail-open semantics (poison entries get `DEL`'d, network errors log + return miss) ensure Redis never blocks a request.

## Scenario 5: Gateway region is down

**What's broken:** `gateway.langwatch.ai`'s primary region (e.g. `us-east-1`) is completely unavailable — AZ failure, DNS outage, power loss, etc.

**What still works:** only what your DNS / LB is configured to fail over to. LangWatch Cloud's `gateway.langwatch.ai` uses Route53 health-check-based failover if you've opted into multi-region; otherwise the single-region deployment is down.

**Operator actions:**

1. **If multi-region:** Route53 automatically moves traffic to the surviving region. VKs that were in L1 on the primary-region pods need to cold-resolve on the secondary — this is fine since the control plane is shared; no cross-region state replication needed.
2. **If single-region self-hosted:** bring the region back. Nothing cleverer is available short of standing up a new gateway deployment in a different region and updating DNS.

**Prevention:** for production self-hosts, run gateway in ≥ 2 regions, share the control plane (or replicate it), and use DNS failover. See [Scaling → Regional placement](/ai-gateway/self-hosting/scaling#regional-placement).

## Scenario 6: Full LangWatch SaaS outage (all regions)

**What's broken:** `app.langwatch.ai` + all gateway regions down. (Extremely rare — but we document the worst case.)

**What still works:** nothing on the LangWatch data path. Your clients get connection failures until you route around.

**Operator actions (for customers who cannot tolerate LangWatch downtime):**

1. Have a pre-planned bypass: keep provider API keys encrypted in your own secrets manager. During full outage, swap your application's `LANGWATCH_ENDPOINT` env var back to provider-native endpoints (`api.openai.com`, `api.anthropic.com`). Restart pods.
2. After LangWatch recovers, swap back. Debits during the bypass window are lost (but so are the traces, spans, and governance — accept the tradeoff).
3. **This is self-hosting's value proposition**: if you run gateway + control-plane in your own infrastructure, this scenario becomes "my cluster is down," which is *your* DR scope, not LangWatch's.

## Backup and recovery posture

**What the gateway needs backed up:**

- Nothing. The gateway is stateless. A backup is its image + its `values.yaml`.

**What the control plane needs backed up:**

- **Postgres**: the `VirtualKey`, `GatewayBudget`, `GatewayBudgetLedger`, `GatewayProviderCredential`, `GatewayAuditLog` tables. Standard Postgres backup (WAL-based preferred).
- **ClickHouse** (if self-hosted): trace ingestion. Losing this means losing historical trace data, not current gateway functionality.
- **Redis** (if self-hosted): nothing — it's a cache, rebuilds from control plane on next miss.

**RTO / RPO expectations:**

- **Gateway RTO**: seconds. `helm upgrade` or `kubectl rollout restart` suffices.
- **Gateway RPO**: zero. No persistent state.
- **Control-plane RTO**: minutes. Depends on your Postgres recovery.
- **Control-plane RPO**: depends on your WAL archival (typically < 1 min).

## Drills

Run these quarterly to verify your DR posture:

1. **Kill a gateway pod** (`kubectl delete pod langwatch-gateway-<random>`). Verify HPA replaces it, in-flight requests complete cleanly, `gateway_draining` gauge went 1 → 0, and no customer-visible errors.
2. **Block egress to the control plane** (temporarily add a `NetworkPolicy` denying egress to `langwatch-app:3000`). Verify existing VKs continue working for > 15 min; new VK creation fails with a clean error. Remove the block. Verify outbox drains.
3. **Block egress to Redis.** Verify latency shifts from ~2 ms to ~30 ms on cold VKs; no errors. Unblock.
4. **Trip a circuit breaker** (`LW_GATEWAY_CIRCUIT_THRESHOLD=2`, send 3 bad requests to a VK without fallback). Verify `502 provider_error` returned; `gateway_circuit_state{credential_id=...}` goes to 2; self-heals 60 s after you stop sending bad requests.
