Every write through the REST API, the CLI, the tRPC routers, or the UI emits a row into `GatewayAuditLog`. Queryable programmatically via `GET /api/gateway/v1/audit` — the enterprise-governance answer to *"who revoked that key on Tuesday?"* without granting database access.

<Note>**Gateway-specific audit viewing lives in the platform's global audit log** at `/settings/audit-log`. Filter by resource type (`virtualKey` / `gatewayBudget` / `gatewayProvider`) to scope to the gateway surface. Full unification — merging `GatewayAuditLog` writes into the main `AuditLog` table so platform admins see every mutation in one query — is a follow-up; until then, gateway writes land in `GatewayAuditLog` and the REST export below is the programmatic path.</Note>

## What's logged

Source of truth: `GatewayAuditAction` enum in `langwatch/prisma/schema.prisma`.

| Action enum value | Actor fields | Target |
|---|---|---|
| `VIRTUAL_KEY_{CREATED, UPDATED, ROTATED, REVOKED, DELETED}` | `actorUserId` (user) or `actor = svc_<projectId>` (service token) | `virtualKey` / `<vk_id>` |
| `BUDGET_{CREATED, UPDATED, DELETED}` | same | `gatewayBudget` / `<budget_id>` (DELETED is the soft-archive) |
| `PROVIDER_BINDING_{CREATED, UPDATED, DELETED}` | same | `gatewayProvider` / `<binding_id>` |
| `CACHE_RULE_{CREATED, UPDATED, DELETED}` | same | `gatewayCacheRule` / `<rule_id>` |

<Info>Guardrail attach/detach events are NOT currently recorded as standalone audit rows — guardrail configuration changes flow through `VIRTUAL_KEY_UPDATED` (the guardrail list is a field on `VirtualKey.config`). Standalone `GUARDRAIL_ATTACHED / DETACHED` actions are a v1.1 follow-up if operators want finer-grained audit.</Info>

Reads are **not** logged — the volume would swamp the ledger without adding governance value. If you need read-access attribution for compliance, use OTel trace logs (every request has `langwatch.principal_id`).

## Viewing in the UI

Go to **/settings/audit-log** (gated on `gatewayLogs:view`) and filter by resource type to see gateway mutations. Six key affordances:

1. **Filter by action** — `VIRTUAL_KEY_CREATED`, `VIRTUAL_KEY_ROTATED`, etc. (UI picker shows these with friendlier labels; the underlying filter sends the enum value.) Useful when investigating "did anyone rotate keys during the incident window?".
2. **Filter by resource type** — `virtualKey` / `gatewayBudget` / `gatewayProvider` / `gatewayCacheRule`. Narrows the view when you're investigating one resource type.
3. **Expand with changed-fields diff summary** — UPDATE rows expand to show a compact top-level diff (e.g. `priority 200 → 300`, `action.ttl 300 → 600` as red/green code badges) above the raw Before/After JSON panels. Common-case scanning is one line per changed field; deep-forensics still has the full JSON below.
4. **CREATE/DELETE events show the full field list** — since those events have a null Before (or null After), the diff summary renders the relevant half as a field list instead of leaving the panel blank for the most common audit event class.
5. **Deep-link from the target column** — clicking the target id opens the resource's detail page (VK detail, budget detail, provider detail). Skips the "which key was that?" lookup.
6. **Deep-link IN from resource detail pages** — every VK detail and budget detail page carries an **Audit history** button that takes you to `/gateway/audit?targetKind=<kind>&targetId=<id>` with a pre-filled filter chip scoped to just that resource. Clear the chip with the `×` icon to re-expand the view. The pattern stays reachable on revoked VKs and archived budgets so forensic investigations start from the current state, not some "active only" fork.

Pagination uses a composite `(createdAt, id)` cursor so batched same-millisecond entries (common when an operator does many mutations from a script) don't dedupe or skip. Load-more button, 50 rows per page.

## Querying programmatically

The same data is available via the tRPC router used by the UI — `gatewayAudit.list`. For REST-first consumers, use a project API token against the LangWatch app's audit export endpoint:

```bash
curl -sS https://app.langwatch.ai/api/gateway/v1/audit \
  -H "Authorization: Bearer $LANGWATCH_API_KEY" \
  -G -d "action=VIRTUAL_KEY_REVOKED" \
     -d "since=2026-04-01T00:00:00Z" \
     -d "limit=100"
```

Response shape:

```json
{
  "data": [
    {
      "id": "aud_01HZ...",
      "action": "VIRTUAL_KEY_REVOKED",
      "actor_user_id": "user_01HZX...",
      "actor_user_email": "alice@example.com",
      "actor": null,
      "target_kind": "virtualKey",
      "target_id": "vk_01HZ...",
      "metadata": { "reason": "rotation cycle" },
      "created_at": "2026-04-18T16:03:12Z"
    }
  ],
  "next_cursor": "2026-04-18T16:03:10Z|aud_01HZ..."
}
```

Actor resolution: `actor_user_id` is set for session / personal-access-token writes (joined from the `User` table so `actor_user_email` comes along). `actor` is set for service-token writes and takes the form `svc_<projectId>`; there's no associated user.

## Retention

Default: 7 years, extendable on request. Stored in the primary app Postgres under `GatewayAuditLog`. The table is partitioned by `createdAt` to keep old-data queries cheap.

## Export

For SIEM / compliance-warehouse pipelines, export via the REST endpoint on a nightly schedule:

```bash
# Pull everything from the last 24h, stream to your SIEM
curl -sS "https://app.langwatch.ai/api/gateway/v1/audit?since=$(date -u -v-1d +%Y-%m-%dT%H:%M:%SZ)" \
  -H "Authorization: Bearer $LANGWATCH_API_KEY" \
  | jq -c '.data[]' \
  | your-siem-ingest
```

Rate-limited to 100 req/min/token; use the cursor for large pulls rather than increasing the `limit` (capped at 500 per request).

## Security considerations

- **Can the audit log be tampered with?** Writes are append-only at the service layer — no UPDATE or DELETE is exposed. Deletions are only possible via direct database access, which is control-plane-operator-only and also audit-logged (via Postgres' built-in `pgaudit` extension, configured in the self-hosted deployment guide).
- **Can an audit entry be forged?** No. Writes happen inside the same transaction as the resource mutation; either both land or neither lands. A caller cannot emit a fake audit entry via REST or CLI.
- **Notifications on security-sensitive writes.** Rotations and revocations generate an organization-wide notification by default (configurable per org). Users with `gatewayLogs:view` also receive an email summary on `VIRTUAL_KEY_REVOKED` for keys they own.
