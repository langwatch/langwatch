The gateway is configured entirely through environment variables — no YAML or config files. Defaults are safe for production; every value below is overridable via Helm values or plain env vars.

## Secrets

These MUST be provided via a Kubernetes Secret (or the equivalent on non-Kubernetes deployments) — never baked into an image.

| Var | Purpose | Shared with control plane? |
|---|---|---|
| `LW_GATEWAY_INTERNAL_SECRET` | HMAC secret used for `/api/internal/gateway/*` calls between gateway and LangWatch control plane | Yes — same value in both |
| `LW_GATEWAY_JWT_SECRET` | HS256 secret for signing the short-lived JWT returned by `/resolve-key` | Yes |
| `LW_VIRTUAL_KEY_PEPPER` | Peppered HMAC secret used to hash virtual-key secrets at rest | Control plane only; gateway never sees it |
| `GATEWAY_REDIS_URL` | Optional L2 cache for resolved VKs, shared across gateway replicas. On HPA scale-out, new pods warm from L2 (~2 ms) instead of round-tripping `/resolve-key` (~30 ms). TTL floored at 30 s; poison entries (decode errors) are `DEL`'d. See [Scaling → Redis L2](/ai-gateway/self-hosting/scaling#redis-l2-optional) | Gateway only |

Rotate `LW_GATEWAY_INTERNAL_SECRET` and `LW_GATEWAY_JWT_SECRET` together via the dual-key overlap flow — the gateway's JWT resolver accepts both `LW_GATEWAY_JWT_SECRET` (current) and `LW_GATEWAY_JWT_SECRET_PREVIOUS` (the retired key) via `jwt.VerificationKeySet` while both are set; see [Secret rotation](#secret-rotation) below.

## Routing

Source of truth: `services/gateway/internal/config/config.go` `Load()`.

| Var | Default | Purpose |
|---|---|---|
| `GATEWAY_LISTEN_ADDR` | `:5563` | Public HTTP listen addr. LangWatch port convention: app=5560, nlp=5561, langevals=5562, gateway=5563 |
| `GATEWAY_ADMIN_ADDR` | `127.0.0.1:6060` | Loopback-bound admin listener (pprof / diagnostics). Binding off-loopback requires `GATEWAY_ADMIN_AUTH_TOKEN` or startup fails |
| `GATEWAY_ADMIN_AUTH_TOKEN` | *(unset)* | Bearer token required for admin listener when bound off-loopback |
| `GATEWAY_LOG_LEVEL` | `info` | `debug` / `info` / `warn` / `error` |
| `GATEWAY_CONTROL_PLANE_URL` | `http://localhost:5560` | Base URL for `/api/internal/gateway/*` calls (HMAC-signed) |
| `GATEWAY_CONTROL_PLANE_TIMEOUT` | `2s` | Per-call timeout for the control-plane HTTP client |
| `GATEWAY_LONG_POLL_TIMEOUT` | `25s` | `/api/internal/gateway/changes` long-poll deadline. Per-org scoped; one goroutine per organization observed on this pod |
| `GATEWAY_NODE_ID` | auto-generated ULID | Unique id for this replica, surfaced in the HMAC `X-LangWatch-Gateway-Node` header |

## Auth cache

| Var | Default | Purpose |
|---|---|---|
| `GATEWAY_CACHE_LRU_SIZE` | `50000` | L1 LRU capacity (per replica) |
| `GATEWAY_REDIS_URL` | *(unset)* | Optional L2 cache URL for JWT-serialised VK bundles. HPA scale-out pods warm from L2 (~2 ms) instead of round-tripping `/resolve-key`. TTL floored at 30 s; poison entries `DEL`'d |
| `GATEWAY_CACHE_REFRESH_INTERVAL` | `60s` | Background refresh interval for warm L1 entries |
| `GATEWAY_CACHE_BOOTSTRAP_ALL_KEYS` | `false` | On startup, pre-warm L1 via `/bootstrap` (all-keys snapshot). Off by default — large orgs opt in |
| `GATEWAY_JWT_REFRESH_THRESHOLD` | `5m` | Refresh JWT when within this window of expiry; backgrounded so the hot path doesn't block |
| `LW_GATEWAY_AUTH_CACHE_SOFT_BUMP` | `5m` | When the control plane is briefly unreachable on refresh, extend the cached entry's soft expiry by this duration on each transport-class failure (network error, dial timeout, 5xx, malformed/unparseable response, JWT verify failure). Auth-class rejections (401/403/404) evict immediately — no grace for known-bad credentials |
| `LW_GATEWAY_AUTH_CACHE_HARD_GRACE` | `30m` | Hard cap on stale-while-error: total grace beyond the JWT's natural exp before forced eviction. Set to `0s` to disable stale-while-error entirely (legacy behaviour: hard-fail at JWT exp on every refresh failure) |

See [Auth Cache Architecture](/ai-gateway/observability#auth-cache-architecture) for the L1/L2 design and [Production runbook → Control-plane outage](/ai-gateway/cookbooks/production-runbook#recipe-8--control-plane-outage--stale-while-error) for the operator playbook.

## Providers + upstream

Provider credentials are managed from the UI (Gateway → Providers), not env vars. The Go gateway uses Bifrost's defaults for upstream timeouts + circuit-breaker windows; operators tune via Bifrost config (the library we embed), not via env vars on the gateway binary. In v1, only the Bifrost pool size + stream buffer are exposed:

| Var | Default | Purpose |
|---|---|---|
| `GATEWAY_BIFROST_POOL_SIZE` | `200` | Bifrost worker pool |
| `GATEWAY_BIFROST_STREAM_BUFFER` | `100` | Per-stream buffer slots |

<Info>Per-provider circuit breaker + upstream timeout tuning env vars (`LW_GATEWAY_UPSTREAM_*`, `LW_GATEWAY_CIRCUIT_*`) are not currently exposed on the gateway binary — Bifrost internal defaults govern. Tracked as a v1.1 configurability follow-up if operators request finer control.</Info>

## Tracing

| Var | Default | Purpose |
|---|---|---|
| `GATEWAY_OTEL_DEFAULT_ENDPOINT` | *(unset; required if you want traces)* | OTLP endpoint the gateway ships all traces to. LangWatch ingest files each trace under the right project via the `langwatch.project_id` span attribute. Self-hosted: point this at your in-cluster OTel collector or LangWatch ingest |
| `GATEWAY_OTEL_BATCH_TIMEOUT` | `5s` | Batch-span-processor flush interval |
| `GATEWAY_OTEL_MAX_QUEUE` | `8192` | Batch-span-processor queue cap |

### Per-project routing (no customer config)

The gateway ALWAYS reads `project_id` from each resolved VK bundle and tags every span with `langwatch.project_id` (plus team, org, principal). Traces are shipped to `GATEWAY_OTEL_DEFAULT_ENDPOINT`; LangWatch ingest uses the span attribute to file each trace under the owning project.

There is no env-var override — LangWatch sells observability, so all traces are routed to the LangWatch pipeline by design.

## Edge protection

### Request body cap

| Var | Default | Purpose |
|---|---|---|
| `GATEWAY_MAX_REQUEST_BODY_BYTES` | `10485760` (10 MiB) | Maximum request body size. Enforced by `httpx.MaxBodyBytes` middleware wired **before** auth: a 1 GB drive-by scan never pressures the pod memory limit (512 Mi default) or the auth cache. Declared `Content-Length` above the cap gets 413 immediately (socket is not drained). Unknown-length bodies (chunked) hit `*http.MaxBytesError` at the cap. 10 MiB was chosen to fit ~3 MB of prompt text + base64 images + headroom, comfortably under a 512 Mi pod cap. Raise for orgs shipping large base64 image arrays; lower for edge deployments with known small payload ceilings |

### HTTP server timeouts

Slowloris-style attacks (byte-trickle, idle-keep-alive flood) are closed by four explicit `http.Server` timeout knobs. The gateway ships safe defaults; operators tune when their LB has different expectations.

| Var | Default | Purpose |
|---|---|---|
| `GATEWAY_SERVER_READ_HEADER_TIMEOUT` | `5s` | Max time from TCP-accept to end-of-headers. Catches TLS-handshake-then-stall probes. Below this is a valid client; above is almost certainly an attacker or a badly-behaved proxy |
| `GATEWAY_SERVER_READ_TIMEOUT` | `60s` | Max time to read the full request body. Stops byte-trickle attacks. 60 s is generous for a 10 MiB legit body on a slow uplink; raise proportionally if you bump `MaxRequestBodyBytes` |
| `GATEWAY_SERVER_IDLE_TIMEOUT` | `120s` | Keep-alive cap. **Invariant**: must be longer than your nginx-ingress `keepalive_timeout` (typical 75 s) so nginx closes first. Otherwise the gateway holds dead sockets after the LB moved on |
| `GATEWAY_SERVER_WRITE_TIMEOUT` | *(deliberately unset)* | Would bound whole-response lifetime, which kills long SSE streams (reasoning models, Claude thinking traces). Per-chunk streaming deadlines live in the dispatcher; whole-response caps belong at your ingress, not here |

All four apply identically to the public listener and the admin listener. See [Helm → Security](/ai-gateway/self-hosting/helm#security) for the values stanza.

The `Security` config struct is intentionally separate from `Upstream` (per-provider budgets) and `Budget` (spend outbox). Edge protection is a distinct concern and its values belong in their own Helm stanza.

## Secret rotation

### `LW_GATEWAY_JWT_SECRET` zero-downtime rotation

Before the dual-key overlap shipped, rotating `LW_GATEWAY_JWT_SECRET` invalidated every outstanding bundle (~15 min TTL) instantly — every in-flight customer request got 401 `invalid_api_key`. No safe rotation short of an announced maintenance window.

Today, the resolver's `Keyfunc` returns `jwt.VerificationKeySet{current, previous}` whenever `LW_GATEWAY_JWT_SECRET_PREVIOUS` is set. `golang-jwt/jwt/v5` natively tries each key in order; current is first so it's the fast path in steady state.

The 4-step zero-downtime rotation flow:

1. **Flip the control-plane signing secret** to the new value. New bundles get signed with `new`.
2. **Update gateway secrets**: `jwtSecretKey=new`, `jwtSecretPreviousKey=old`. Deploy.
3. **Rolling restart** picks up both keys. Pre-rotation bundles (signed with `old`) keep verifying via the previous-key fallback. New bundles sign+verify with `new`.
4. **After the longest pre-rotation bundle TTL has elapsed** (~15 min default), remove `jwtSecretPreviousKey` entirely. Rolling restart again. Strict-mode resumes.

The gateway emits a `jwt_secret_rotation_active` WARN log on boot whenever `JWTSecretPrevious` is set, with explicit "remove once ~15m elapsed" guidance. Accepting a retired key forever is a posture regression — the WARN keeps operators honest.

| Var | Default | Purpose |
|---|---|---|
| `LW_GATEWAY_JWT_SECRET_PREVIOUS` | *(empty)* | Retired JWT signing secret. Set during step 2 of the rotation; remove after step 4. Empty = strict mode (default / steady state) |

See [Helm → Secret rotation](/ai-gateway/self-hosting/helm#secret-rotation) for the `values.yaml` shape.

### `LW_GATEWAY_INTERNAL_SECRET` rotation

The HMAC signing secret for `/api/internal/gateway/*` calls uses the same dual-secret pattern — `LW_GATEWAY_INTERNAL_SECRET_PREVIOUS` is accepted alongside the current value during the rolling window, then removed. The mechanics are identical; both secrets can be rotated in the same operation.

## Graceful shutdown

| Var | Default | Purpose |
|---|---|---|
| `GATEWAY_SHUTDOWN_PRE_DRAIN_WAIT` | `5s` | After SIGTERM, the gateway flips `/readyz` to 503 `draining` + the `gateway_draining` gauge to 1, then sleeps this long before calling `server.Shutdown`. Gives the LB endpoint controller (nginx-ingress / kube-proxy) time to remove the pod from the service endpoints. EKS observed propagation is 3-4 s; 5 s leaves margin |
| `GATEWAY_SHUTDOWN_TIMEOUT` | `15s` | Hard cap on the `server.Shutdown` blocking wait. In-flight handlers get this long to finish; on expiry the server force-closes. Streaming handlers should respect request context cancellation to drain within this window |

The `Shutdown` config struct is intentionally separate from `Security` (edge protection) — shutdown is a lifecycle concern, not a request-pipeline one. Operators should tune the two knobs together: `terminationGracePeriodSeconds` on the pod spec must be ≥ `preDrainWait + timeout + slack` (default chart renders 30 s = 5 + 15 + 10 slack).

During drain: `/livez` and `/startupz` are deliberately unaffected (pod is alive and started, just refusing new work). Liveness staying 200 prevents kubelet from short-circuiting the drain via a `livenessProbe` failure. `/readyz` is the LB signal.

## Startup gate

| Var | Default | Purpose |
|---|---|---|
| `GATEWAY_STARTUP_NETCHECK_HOSTS` | *(empty)* | Comma-separated `host:port` list. On startup, the gateway resolves DNS + dials TCP for each entry; any failure causes `/startupz` to stay red, which makes k8s recycle the pod **before** `MarkStarted` fires. Designed to catch `NetworkPolicy` misconfig at deploy time, not first customer request. DNS-class and TCP-class failures are distinguished in the error string so operators can pinpoint which egress rule is wrong. See [Helm → Startup network check](/ai-gateway/self-hosting/helm#startup-network-check) |
| `GATEWAY_STARTUP_NETCHECK_TIMEOUT` | `2s` | Per-host timeout (DNS + TCP combined). Stop-on-first-failure — the probe exits as soon as one host is unreachable |

## Admin / operator endpoints

| Var | Default | Purpose |
|---|---|---|
| `GATEWAY_ADMIN_ADDR` | `127.0.0.1:6060` | Listener exposing Go `net/http/pprof` for heap / goroutine / profile / allocs / mutex / block dumps. Default is loopback-only — reachable exclusively via `kubectl port-forward`. Set to empty string to disable the listener entirely. Set to `0.0.0.0:6060` (or similar) to listen non-loopback; **in that case a bearer token is required** |
| `GATEWAY_ADMIN_AUTH_TOKEN` | *(empty)* | Bearer token guarding the admin listener. Compared via `crypto/subtle.ConstantTimeCompare`. Required whenever `GATEWAY_ADMIN_ADDR` is non-loopback and `GATEWAY_ALLOW_INSECURE` is unset; the gateway refuses to start otherwise with a clear error naming both env vars |
| `GATEWAY_ALLOW_INSECURE` | `false` | Escape hatch to run a non-loopback admin listener without a bearer token. Exists for test harnesses — never set this in production |

Three deployment postures for the admin listener:

**1. Disabled.** Set `GATEWAY_ADMIN_ADDR=""`. No pprof at all. Pick this for compliance-regulated deployments where live profiling is not permitted.

**2. Loopback-only (default).** `GATEWAY_ADMIN_ADDR=127.0.0.1:6060`, no token. Operator access is via `kubectl port-forward` through the Kubernetes API server, which is auditable in cluster audit logs. Bearer token is optional and works as a second defence layer.

**3. Bearer-gated non-loopback.** `GATEWAY_ADMIN_ADDR=0.0.0.0:6060` + `GATEWAY_ADMIN_AUTH_TOKEN` in a Kubernetes Secret. Pick this for non-k8s deployments (systemd / Nomad / plain Docker) where port-forward isn't available, or when you want direct access through your cluster's internal LB.

See [Helm → Admin listener](/ai-gateway/self-hosting/helm#admin-listener) for the Helm values shape.

Operator workflow (loopback posture, safe-by-default because the port isn't in any Service):

```bash
# Forward the admin port from a specific pod
kubectl port-forward -n langwatch deploy/langwatch-gateway 6060:6060

# Live goroutine dump when p99 spikes
go tool pprof -svg http://localhost:6060/debug/pprof/goroutine > /tmp/goroutines.svg

# 30-second CPU profile under production load
go tool pprof http://localhost:6060/debug/pprof/profile?seconds=30
```

See [Cookbook: Production runbook](/ai-gateway/cookbooks/production-runbook) for full diagnostic recipes.

## Operational defaults worth knowing

- The gateway preloads every VK's config on startup (via `/bootstrap`) if `LW_GATEWAY_AUTH_CACHE_BOOTSTRAP=true`. This adds ~1–3 s to startup for a 50K-key tenant. Disable it for horizontally-sharded deployments where each replica only serves a subset.
- `SIGTERM` handling: 30 s drain, refuse new requests with 503, finish in-flight streams. Matches Kubernetes `preStop` + `terminationGracePeriodSeconds: 35`.
- `PROTECTED_ENDPOINTS` (internal): the `/healthz` / `/readyz` / `/metrics` endpoints never require auth and are not rate-limited — give your scraper direct access.

See [Health Checks](/ai-gateway/self-hosting/health-checks) for what each probe validates and [Scaling](/ai-gateway/self-hosting/scaling) for HPA and capacity planning.

## Request body cap

| Var | Default | Purpose |
|---|---|---|
| `SERVER_MAX_REQUEST_BODY_BYTES` | `33554432` (32 MiB) | Per-request body cap enforced at the edge before auth or upstream dispatch. Sized for 1 M-context LLM workloads (Gemini 1.5 Pro full-context with images can exceed 10 MiB; Claude with long doc context 5–8 MiB; coding-agent sessions with rich tool schemas 4–6 MiB). Returns `413 payload_too_large` when exceeded |

Public-edge deployments tightening DDoS protection can lower this; enterprise deployments processing full-context multimodal payloads can raise it. The gateway also applies a per-endpoint peek window (32 KiB on `/v1/chat/completions` and `/v1/messages`, 256 KiB on `/v1/responses` to find `stream:true` past large tool/instruction blocks); peek size is internal and not env-tunable today.

## Debug / development env knobs

These are gated env vars used during dev / triage. Never enable in production — they emit verbose data including potentially sensitive prompt content.

| Var | Default | Purpose |
|---|---|---|
| `LW_LOG_MESSAGE_BODY` | unset | Set to `1` to log a peek of every inbound `/v1/messages` and `/v1/responses` request body at INFO level. Useful for diagnosing CLI-shape-specific provider rejections (e.g. exact body structure that trips Anthropic's edge HTML response) |
| `LW_GATEWAY_OUTBOUND_PROXY` | unset | Set to an `http://host:port` URL to route all outbound provider HTTP traffic through a logging proxy (mitmproxy, mitmdump, etc.). Bifrost's `ProxyConfig` is wired from this; covers OpenAI / Anthropic / Gemini / Azure (Bedrock uses the AWS SDK and bypasses HTTP proxy). Use to capture exact outbound URL + headers + body for cache-key-hash debugging or upstream-quirk diagnosis |
| `FEATURE_FLAG_FORCE_ENABLE` | unset | Comma-separated list of feature flags to force-enable, bypassing PostHog. Most common in dev: `release_ui_ai_gateway_menu_enabled` to make the AI Gateway menu visible without a PostHog distinct-id match. Read by the control plane on boot — restart `pnpm dev` after changing |

Worked example — capturing outbound provider traffic to debug a cache-hit hash mismatch:

```bash
mitmdump -p 8888 --ssl-insecure &
export LW_GATEWAY_OUTBOUND_PROXY=http://localhost:8888
export LW_LOG_MESSAGE_BODY=1
make service svc=aigateway   # restart picks up both env vars
```

Replay the failing request via your client; the mitmdump session shows the exact bytes the gateway sends to `api.anthropic.com` (or any other provider), and the gateway log line `peek=...` shows what the client sent in. Diff side-by-side to localise body-mutation bugs (this exact recipe found a cache-rule wire-DTO matcher bug — `disable-cache-evals` rule was matching every VK and stripping `cache_control` from outbound bodies, narrowed via 40-byte byte-diff).
