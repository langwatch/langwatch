When you run LangWatch alongside another OTel-based SDK (for example, an APM
tool), both SDKs may hook into the same global `TracerProvider`. This causes
**cross-contamination** — LLM traces appear in the other tool and application
traces appear in LangWatch.

This guide shows how to configure LangWatch for each SDK. The approach differs
between Python (true provider isolation) and TypeScript (coexistence on the
shared provider with optional filtering).

## The Problem

OpenTelemetry has a single global `TracerProvider`. When two SDKs both connect to
it, every span flows to every exporter:

```
┌─────────────────────────────────────────────┐
│          Global TracerProvider               │
│                                             │
│  ┌─────────────┐    ┌────────────────────┐  │
│  │ Other SDK's │    │ LangWatch's        │  │
│  │ Exporter    │    │ Exporter           │  │
│  └──────┬──────┘    └─────────┬──────────┘  │
│         │                     │             │
│    ALL spans              ALL spans         │
└─────────────────────────────────────────────┘
```

Both exporters receive all spans — HTTP requests, database queries, LLM calls,
and everything else.

## Python SDK — Provider Isolation

The Python SDK supports **true provider isolation**. Pass a dedicated
`TracerProvider` to `langwatch.setup()` — LangWatch creates spans on its own
provider while the other SDK keeps the global one. No cross-contamination in
either direction.

```python
from opentelemetry.sdk.trace import TracerProvider
import langwatch

# 1. The other OTel SDK initializes first (sets the global provider)

# 2. Create a dedicated provider for LangWatch
langwatch_provider = TracerProvider()

# 3. LangWatch uses this provider exclusively
langwatch.setup(tracer_provider=langwatch_provider)
```

With a dedicated provider:
- LangWatch only receives spans created through its own tracing API (`langwatch.trace()`)
- The other SDK only receives spans on the global provider
- Neither SDK sees the other's spans

### Using span_exclude_rules (alternative)

If you prefer to share the global provider instead of isolating, you can filter
out unwanted spans from LangWatch using exclude rules:

```python
from langwatch.domain import SpanProcessingExcludeRule
import langwatch

langwatch.setup(
    span_exclude_rules=[
        SpanProcessingExcludeRule(
            field_name="span_name",
            match_operation="starts_with",
            match_value="GET",
        ),
        SpanProcessingExcludeRule(
            field_name="span_name",
            match_operation="starts_with",
            match_value="POST",
        ),
    ],
)
```

This approach is a name-based blocklist and may not catch every non-LLM span.
The dedicated provider approach above is more reliable.

## TypeScript SDK — Coexistence

The TypeScript SDK uses `advanced.attachToExistingProvider` to add LangWatch's
span processor to the existing global provider. This is **coexistence, not
isolation** — LangWatch receives all spans from the shared provider, including
application spans from the other SDK.

```typescript
// 1. The other OTel SDK initializes first (sets the global provider)
import "./other-sdk-init";

// 2. LangWatch attaches to the existing provider
import { setupObservability } from "langwatch/observability/node";

setupObservability({
  langwatch: {
    apiKey: process.env.LANGWATCH_API_KEY,
  },
  advanced: {
    attachToExistingProvider: true,
  },
});
```

Without `attachToExistingProvider`, LangWatch detects the existing provider and
returns a no-op — no spans are captured. With it enabled, LangWatch adds its
span processor to the existing provider so spans are exported to LangWatch.

<Note>
Since both SDKs share the same provider, LangWatch will receive **all** spans —
not just LLM spans. To scope LangWatch to only LLM-related spans, configure
filtering on the `LangWatchTraceExporter` (see below).
</Note>

### Filtering to LLM-only spans

To prevent non-LLM spans from reaching LangWatch, use the
`LangWatchTraceExporter` filter options. See the
[TypeScript SDK reference](/integration/typescript/reference) for filter
configuration, including presets like `vercelAIOnly` and `excludeHttpRequests`.

## What about the other direction?

**Python:** The dedicated provider gives full isolation — LLM spans never reach
the other SDK because they're created on a separate provider.

**TypeScript:** Since both SDKs share the global provider, LLM spans will also
reach the other SDK's exporter. To prevent this, configure the other SDK's own
span filtering mechanism.
