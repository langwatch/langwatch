When you are choosing between two prompts, two models, or two agent configurations, absolute evaluator scores can be hard to calibrate. A score of `0.78` does not always tell you whether the new variant is actually better than the current one for your users.

**Pairwise Compare** is built for that decision. It shows an LLM judge both candidate outputs for the same dataset row and asks which one is better: Variant A, Variant B, or a tie. The result is a head-to-head verdict for each row plus an aggregate win/tie summary across the run.

## Why use Pairwise Compare?

Pairwise comparison is useful because the judge only has to answer a relative question: "which of these two outputs is better for this input?" That is often easier and more stable than asking for an absolute score in isolation.

Use it when you want to:

- **Choose a winner between two finalists.** Compare a baseline prompt against a proposed rewrite, a current model against a cheaper model, or an old agent configuration against a new one.
- **Evaluate subjective quality.** Tone, helpfulness, concision, formatting, and reasoning quality often have no single golden answer, but one output can still be clearly better than the other.
- **Review row-level tradeoffs.** Pairwise results show where Variant A wins, where Variant B wins, and where the judge sees no meaningful difference.
- **Reduce position bias.** With `swap_and_confirm`, LangWatch can judge A vs B and then B vs A, only declaring a winner when both calls agree.
- **Factor in operational metrics.** Include candidate cost or duration in the judge context when you want the better answer to also account for latency or spend.

## When to use it

Pairwise Compare is the right fit when the decision is truly between two alternatives:

- You are reviewing a prompt change before promoting it.
- You are testing whether a smaller or cheaper model preserves answer quality.
- You are comparing two retrieval, tool-use, or agent policies on the same dataset.
- You have a golden answer, but want to judge which candidate follows it better.
- You do not have a golden answer, but can describe what "better" means in judge instructions.
- You want a human-reviewable explanation for each win, loss, or tie.

It is not the best first tool when:

- You only have one target to score. Use [Expected Answer Evaluation](/evaluations/experiments/ui/answer-correctness) or [LLM-as-a-Judge](/evaluations/experiments/ui/llm-as-a-judge).
- You need a calibrated numeric metric such as exact match, retrieval recall, toxicity, or latency.
- You want to rank many candidates in one pass. Pairwise is for two variants; n-way comparison is the future workflow for that.

## Setting it up

1. Open the [Experiments Workbench](https://app.langwatch.ai/@project/evaluations).
2. Add the two prompts, agents, models, or other targets you want to compare.
3. Add **Pairwise Compare** as a comparison target.
4. Select **Variant A** and **Variant B** from the existing targets.
5. If your dataset has a reference answer, enable **Has golden answer** and select the golden field.
6. If there is no reference answer, disable **Has golden answer** and use judge instructions to define the preference criteria.
7. Run the experiment. Pairwise Compare runs after both candidate variants have produced outputs for the row.

You can compare two variants even when they have the same display name. LangWatch numbers same-name variants, for example `bot (1)` and `bot (2)`, so the scoreboard and row verdicts stay readable without guessing whether the difference is model, temperature, prompt text, or another configuration detail.

## How the verdict works

Pairwise Compare can use `swap_and_confirm` to reduce first-position bias:

1. Judge call one compares Candidate A vs Candidate B.
2. Judge call two repeats the comparison with the candidates swapped.
3. If both calls agree on the same winner, that candidate wins.
4. If the two calls disagree, the evaluator returns a tie instead of forcing a winner.

This means `swap_and_confirm` uses two judge calls per row. The setting is useful when you want the result to be less sensitive to which candidate appears first in the judge prompt.

The verdict label is stored as Variant A, Variant B, or tie. The details include the judge reasoning so you can inspect why a row went one way.

## Golden answer vs direct preference

When `has_golden_answer` is enabled, the evaluator judges both candidates relative to the reference answer. When it is disabled, the evaluator compares Candidate A and Candidate B directly on their own merits.

Use a golden answer when correctness is the primary question. Use direct preference when the task is open-ended and you care about qualities such as tone, completeness, helpfulness, structure, or policy adherence.

## Reading the results

Pairwise Compare is designed so the result column contains the information you need to review the judgment without opening a separate panel:

- **Who won:** each row starts with a winner badge, or a tie badge when neither candidate is clearly better.
- **What won:** the cell shows the winning variant's actual output so you can immediately inspect the answer the judge preferred.
- **Why it won:** the judge reasoning appears below the output, including the comparison logic that led to the verdict.

The results page also includes a Pairwise Compare win-rate chart alongside the cost and latency charts. It shows how many rows Variant A won, how many rows Variant B won, and how many rows tied.

In the workbench, the pairwise column header keeps a scoreboard tally across the current run. For example, you might see `Variant A wins 12 · 2 ties`. Hover the header to see the full breakdown for both variants and ties.

If a verdict references a variant and you want to inspect its source output, click the variant name in the verdict. LangWatch highlights the corresponding source column header so you can map the judgment back to the exact candidate output.

Review the aggregate result and the row-level explanations together:

- A large win margin usually indicates a clear preference.
- A high tie count often means the two variants are equivalent for the dataset or the judge instructions are not specific enough.
- Split wins are useful signal: inspect the rows each variant wins to understand which use cases improved or regressed.

## Settings

| Setting | Default | Description |
| ------- | ------- | ----------- |
| `model` | — | The judge model. |
| `has_golden_answer` | `true` | Compare each candidate against a reference answer. Disable it to compare the candidates directly. |
| `swap_and_confirm` | `true` | Run two judge calls with the candidates swapped, and return a tie if the calls disagree. |
| `allow_tie` | `true` | Allow the judge to return a tie when the candidates are effectively equivalent. |
| `include_metrics` | `[]` | Include per-candidate metrics such as cost or duration in the judge context. |
| `prompt` | — | Optional custom judge instructions. |

## Pairwise vs future n-way comparison

Pairwise Compare answers one focused question: **which of these two candidates is better?**

The planned n-way comparison flow is for a different question: **which candidate should win among three or more options?** That workflow will be better suited for ranking multiple generated variants, selecting the best candidate from a broader search, or comparing several models at once.

Until then, use Pairwise Compare for deliberate A/B decisions. If you have several candidates today, narrow them with other metrics first, then run Pairwise Compare on the finalists or on the specific challenger-vs-baseline decisions that matter most.

## Using it from the SDK

Pairwise Compare is a regular built-in evaluator under the hood: `langevals/pairwise_compare`. You can call it from the SDK the same way you call any other evaluator. See [Experiments via SDK](/evaluations/experiments/sdk#pairwise-compare) for a concrete example.

If you run your own pairwise judge locally, you can still publish its score, label, and details with the SDK. Those custom results render as standard evaluator results today. The dedicated pairwise column visualization is driven by LangWatch's Pairwise Compare configuration.
