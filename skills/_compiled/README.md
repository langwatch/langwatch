# Compiled skill outputs

Generated from the canonical `skills/*/SKILL.mdx` sources (partials inlined).
Two different outputs with two different lifecycles — see `skills/README.md`
for the full pipeline.

## `*.txt` — copy-paste prompts (gitignored)

- `<skill>.platform.txt` — carries a `{{LANGWATCH_API_KEY}}` placeholder, for
  platform pages that substitute the user's real key.
- `<skill>.docs.txt` / `recipes-<name>.docs.txt` — tell the agent to ask the
  user for their API key, for documentation pages.

These are NOT committed. The docs pipeline regenerates them on the fly:
`docs/scripts/sync-prompts.sh` runs `generate.sh`, then folds the `.txt` files
into `docs/snippets/prompts-data.jsx` (which IS committed and freshness-checked
by docs-ci).

## `native/` — opencode skills for Langy (committed)

One `<slug>/SKILL.md` per published skill (recipes flattened), generated by
`skills/_compiler/native.ts`. `Dockerfile.langyagent` copies this tree into the
Go manager's embed dir at image build, so **the checked-in content is exactly
what the in-product assistant ships with**. `skills/_tests/native-skills.test.ts`
fails when this tree drifts from the sources.

## Regenerating

After modifying any `SKILL.mdx` or `_shared/*.mdx` file:

```bash
bash skills/_compiled/generate.sh
```

Commit the resulting `native/` changes together with the source change.

## Testing

Compiled outputs are the same content as the sources with `_shared/` partials
inlined — the structural tests (`native-skills.test.ts`, `publish-sync.test.ts`)
verify that mechanically, and a skill's dogfood scenario test covers the
content's quality for every rendering of it.
